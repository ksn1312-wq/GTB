Strengths
Clean two-endpoint symmetry GET baseline / POST personalized share one response schema, enabling incremental partner adoption without divergent integration paths.
Correct cache key design Personalized cache keyed on preference hash + engine + data version prevents stale cross-preference collisions.
Thorough error model 410 hotel_delisted and 422 insufficient_data are non-obvious codes that show real operational maturity and reduce partner support burden.
Realistic SLOs Sub-300 ms cached and sub-800 ms computed are achievable with Redis and are specific enough to gate milestone acceptance.
Well-sequenced workstreams 11 workstreams map clearly to 5 milestones; engine extraction before endpoint build is the correct dependency order.
Scoped API keys scores:read scope and optional IP allowlists are good defaults for a B2B surface targeting insurers and assisted-living operators.
Risks
R-001 High

Baseline scoring math not locked before M1. Partners will observe unexplained score drift across data refreshes if default preference weighting is unstable.

Action: Close OQ #1 before M1 kickoff; freeze engine default weights and version them in the OpenAPI schema.
R-002 High

No cache invalidation policy defined. The 99.9% availability SLA may be breached without specifying stale-on-miss vs. fail-open behaviour before the Redis layer is built.

Action: Define invalidation policy and stale-serve timeout in the design doc prior to M2.
R-003 High

agodaHotelId exposed in the public B2B schema leaks data sourcing and creates versioning obligations if the Agoda partnership changes.

Action: Resolve OQ #5; move field behind include= param or a scope flag before private beta.
R-004 Medium

include=raw lacks a contractual definition. Exposing raw scores without a spec creates support debt and partner lock-in on internal implementation details.

Action: Define raw semantics or cut from v1; add to OpenAPI schema with a clear description.
R-005 Medium

No webhook or event model planned. OTAs caching scores locally will poll at high frequency, inflating quota consumption and masking data staleness.

Action: Stub a score-changed event in the v1.1 plan now to prevent a painful retrofit.
R-006 Medium

Preference enum published without a governance process. Partners build integrations against exact strings; any rename breaks them silently.

Action: Version the enum in OpenAPI; document the deprecation policy before private beta.
R-007 Medium

Rate limit aggregation unspecified — per-key vs. per-account. Multi-key partners may bypass the intended 10k req/day cap.

Action: Resolve before auth middleware build (workstream 2); document in the partner-facing quota policy.
R-008 Low

No API versioning sunset policy mentioned. Partners need a migration guarantee (min 12 months) before committing integration effort.

Action: Add a migration policy section to the API contract before public beta.
Assumptions
A-001 High

The internal scoring engine can be extracted cleanly without significant refactoring. If tightly coupled, M1 timeline is immediately at risk.

Action: Run an extraction spike in week 1; validate before M1 milestone date is set.
A-002 High

Pricing model (OQ #3) will be resolved before private beta. Beta partner recruitment is blocked without knowing what tiers and quotas they are signing up for.

Action: Close OQ #3 before M3; pricing must precede partner agreement drafts.
A-003 Medium

Data freshness SLA (OQ #4) is achievable at the proposed cache TTL. No data refresh cadence is documented in the proposal.

Action: Define refresh cadence and include it in the SLO section before M2.
A-004 Medium

The 9-preference enum is stable for the entire v1 lifecycle. Any post-launch rename or addition breaks all partner integrations using enum validation.

Action: Freeze enum at M2; communicate the lock to all product stakeholders.
A-005 Medium

P95 < 800 ms for the personalized endpoint is achievable without specifying Redis cluster sizing or CDN configuration details.

Action: Include load test benchmarks (workstream 9) against the target percentile before M4.
A-006 Low

TLS 1.2+ is sufficient. Some regulated partners — insurers, assisted-living operators — may contractually require TLS 1.3.

Action: Survey beta partner security requirements during M3; add TLS 1.3 support if required for GA.
Dependencies
D-001 High

Baseline and personalized endpoint builds (workstreams 3 & 4) cannot start until engine extraction (workstream 1) is complete. This is the critical path with no documented buffer.

Action: Sequence workstreams explicitly in the project plan; do not parallelize 3 or 4 with 1.
D-002 Medium

OpenAPI docs (workstream 7) depend on schema, error model, and enum being frozen. Publishing draft docs against a moving schema gives partners false confidence during private beta.

Action: Gate workstream 7 start on schema freeze at M2; do not publish draft docs before that point.
D-003 Medium

M3 private beta partner recruitment depends on pricing model (OQ #3) and Agoda field decision (OQ #5) being resolved. Both are currently open with no owner or deadline.

Action: Assign owners and resolution milestones to all OQs; close OQ #3 and #5 before M2.
D-004 Low

Load testing (workstream 9) depends on Redis caching (workstream 6) being production-configured. Testing against a cold cache will produce misleading latency data.

Action: Ensure workstream 6 is complete and cluster-sized before workstream 9 begins.
